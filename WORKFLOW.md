# 🔄 KrishiDisha - Complete Workflow & Architecture Guide

This document provides a highly detailed, step-by-step breakdown of the KrishiDisha platform. It explains how features connect, how the frontend interacts with the backend, the page-to-page navigation flow, and an in-depth breakdown of all 8 roles.

---

## 🏗️ 1. Architecture: Frontend ↔ Backend ↔ Database

KrishiDisha uses a tightly coupled **Monolithic Architecture** (PHP + MySQL).

* **Frontend (UI):** Built with HTML5, CSS3, and JavaScript. The frontend defines the visual interface and collects user inputs via `<form>` tags.
* **Backend (PHP Engine):** When a frontend form is submitted (e.g., clicking "Order"), an HTTP POST request is sent to the PHP file. PHP intercepts this `$_POST` data, runs validation, and executes business logic.
* **Database (MySQL):** PHP uses **PDO (PHP Data Objects)** to execute SQL queries. It uses `prepare()` statements to prevent SQL injection. Once data is retrieved or inserted, PHP dynamically generates the updated HTML frontend and sends it back to the user's browser.
* **Function-to-Function Connection:** Functions like `isLoggedIn()`, `requireAuth()`, and `currentRole()` (located in `includes/auth_check.php`) are included at the very top of every protected page. They intercept the page load, check the PHP `$_SESSION` variables, and either grant access or redirect the user.

## 🌍 2. Page-to-Page Navigation Flow

How does a user travel through the application?

1. **Landing Page (`index.php`):** The public entry point. It contains marketing information, statistics, and calls-to-action (CTAs).
2. **Authentication Flow (`auth/login.php` & `auth/register.php`):**
   * If a user clicks "Login" from `index.php`, they go to `login.php`.
   * Upon successful login, the backend `login.php` script assigns `$_SESSION['role']` and executes a `header("Location: ...")` redirect to the user's specific dashboard.
3. **Dashboards (`<role>/dashboard.php`):** The central hub for a logged-in user. From here, users access their specific tools via the `includes/sidebar.php` navigation menu.
4. **Shared Modules (`modules/*.php`):** Intelligence tools (Encyclopedia, Recommender, Disease Detection) are accessible from the sidebar. These pages do not require specific roles, meaning any user can navigate to them from their dashboard.

## 🔗 3. Feature-to-Feature Connections (Workflows)

Features in KrishiDisha are deeply interconnected. An action by one role triggers a reaction in another role's dashboard.

### A. The Direct-to-Consumer (D2C) Marketplace Workflow
* **Feature 1 (Farmer Produce):** Farmer submits a form on `farmer/produce.php`. Backend PHP executes an `INSERT INTO PRODUCT`.
* **Connection:** The `user/marketplace.php` page runs a `SELECT * FROM PRODUCT WHERE status='available'`. The farmer's new item instantly appears here.
* **Feature 2 (User Ordering):** General User clicks "Order". The form sends a POST request back to `marketplace.php`. 
* **Connection (The Transaction):** PHP initiates a database transaction. It inserts into `ORDER`, deducts `quantity_kg` from `PRODUCT`, inserts into `PAYMENT`, and calculates a 5% cut to insert into `ADMIN_COMMISSION`.
* **Feature 3 (Admin Monitoring):** The Admin visits `admin/commissions.php`. The backend queries the `ADMIN_COMMISSION` table and displays the 5% cut generated by the user's order.

### B. The Unified Consultation & Chat Workflow
* **Feature 1 (Provider Listing):** Experts and Guides are approved by the Admin. Their data exists in the `EXPERT` and `GUIDE` tables.
* **Feature 2 (Booking):** Any user visits `modules/book_consultation.php`. The frontend displays a dropdown populated by querying both `EXPERT` and `GUIDE` tables. User submits booking. Backend inserts into `CONSULTATION`.
* **Connection:** The Expert/Guide visits their `sessions.php` page. The backend queries `CONSULTATION` where `provider_id` matches their ID. They click "Accept", updating the status.
* **Feature 3 (Live Chat):** Both parties click the "Chat" button. They are routed to `modules/consultation_chat.php?id=X`. Frontend provides a text area. Submitting a message triggers an `INSERT INTO CONSULTATION_MESSAGE`. When the page reloads, the backend fetches all messages for that ID and the frontend renders them as chat bubbles.

### C. The Agri-Tourism & Food Workflow
* **Feature 1 (Farm Listing):** Farmer lists land in `farmer/farmland.php` -> updates `FARM_TOUR`.
* **Feature 2 (Tour Booking):** Tourist books tour in `tourist/tours.php` -> updates `TOUR_BOOKING`.
* **Feature 3 (Guide Hire):** Tourist optionally hires a guide. The Guide's `guide/bookings.php` page fetches `TOUR_BOOKING` records assigned to them.
* **Feature 4 (Food Order):** Cook adds recipes to `RECIPE`. Tourist orders food in `tourist/food_orders.php` -> updates `FOOD_ORDER`. Cook fulfills it in `cook/orders.php`.

### D. The Intelligence Modules Workflow (Shared Features)
These modules are accessible by all roles and provide dynamic, data-driven insights.

* **Feature 1 (Crop Encyclopedia):** User visits `modules/encyclopedia.php`. The frontend presents filter dropdowns (Season, Category). Submitting the filter reloads the page with `$_GET` parameters. 
* **Connection:** PHP queries the `CROP` table based on the GET parameters. Clicking "Nutrition Info" opens a Bootstrap modal. This triggers a backend query joining `CROP_VITAMIN` and `VITAMIN` to fetch the exact milligram (mg/mcg) values for that specific crop.
* **Feature 2 (Disease Detection):** User visits `modules/disease.php`. The frontend has a live search bar.
* **Connection:** As the user searches, PHP queries the `DISEASE` table joined with `CROP_DISEASE` and `CROP`. It outputs the Symptoms, Organic Solutions, and Chemical Solutions for each plant disease.
* **Feature 3 (Crop Recommender):** User visits `modules/recommend.php`. They choose between "By Region" or "By Nutrition".
* **Connection:** If "By Region" is selected, the frontend form sends a POST request with the region name (e.g., "Sylhet"). The backend queries `REGION_CROP` joined with `CROP`, sorts by `suitability_score DESC`, and returns the best crops. If "By Nutrition" is selected, it queries `CROP_VITAMIN` to find crops richest in the chosen vitamin.
* **Feature 4 (Nutrition Retention):** User visits `modules/nutrition.php`. They select a crop and a cooking method.
* **Connection:** The backend queries `NUTRIENT_RETENTION` linking the crop and method. The frontend uses JavaScript to animate progress bars visually showing the percentage of vitamins retained (e.g., Boiling retains 60% of Vitamin C).
* **Feature 5 (Profit Calculator):** User visits `modules/calculator.php`.
* **Connection:** PHP embeds the current market prices from the `PRODUCT` table into a hidden JavaScript array. When the user inputs their land size on the frontend, JavaScript instantly multiplies the acreage by the embedded market price, deducts standard costs, and dynamically updates the DOM to show the Estimated Net Profit without refreshing the page.

### E. The Admin & Approval Workflow
* **Feature 1 (Registration):** A user registers as an Expert, Guide, or Cook on `auth/register.php`. The backend assigns them a `status` of `'pending'`.
* **Connection (The Block):** If they try to log in, `auth_check.php` sees their status is pending and throws an error, blocking access to the platform.
* **Feature 2 (Admin Dashboard):** Admin logs in and visits `admin/approvals.php`. The backend queries `SELECT * FROM USER WHERE status='pending'`.
* **Connection (The Approval):** Admin clicks the "Approve" button. This sends a GET request to the backend, which executes an `UPDATE USER SET status='approved' WHERE id=?`.
* **Feature 3 (Access Granted):** The user can now successfully log in and access their role-specific dashboard.

## 🎭 4. The 8 Roles: Complete Breakdown

Every role has a distinct UI, interacts with specific backend logic, and manipulates specific database tables.

### 1. 🛡️ Admin
* **Frontend:** `admin/dashboard.php`, `admin/approvals.php`, `admin/commissions.php`.
* **Backend Functionality:** Monitors platform activity. Has buttons to "Approve" or "Reject" pending users.
* **Database Connection:** Queries the `USER` table for `status='pending'`. Clicking approve runs an `UPDATE USER SET status='approved'`. Queries `ADMIN_COMMISSION` to track platform revenue.

### 2. 🧑‍🌾 Farmer
* **Frontend:** `farmer/dashboard.php`, `farmer/produce.php`, `farmer/farmland.php`.
* **Backend Functionality:** Can list physical crops for sale and list their physical land for tourist visits.
* **Database Connection:** 
  * `produce.php` inserts data into `PRODUCT`. 
  * `farmland.php` inserts data into `FARM_TOUR`. 
  * Dashboard queries `COUNT(*)` from `PRODUCT` and `FARM_TOUR` to show statistics.

### 3. 👤 General User
* **Frontend:** `user/dashboard.php`, `user/marketplace.php`, `user/nutrition.php`.
* **Backend Functionality:** Browses the marketplace to buy crops directly from farmers. Accesses nutritional guides.
* **Database Connection:** `marketplace.php` queries the `PRODUCT` table. Submitting an order form inserts into the `ORDER` and `PAYMENT` tables.

### 4. 🧳 Tourist
* **Frontend:** `tourist/dashboard.php`, `tourist/tours.php`, `tourist/food_orders.php`.
* **Backend Functionality:** Books farm tours listed by Farmers, hires Guides, and orders food from Cooks.
* **Database Connection:** Submitting a tour booking form inserts into `TOUR_BOOKING`. Submitting a food order inserts into `FOOD_ORDER`.

### 5. 🧑‍🍳 Cook
* **Frontend:** `cook/dashboard.php`, `cook/recipes.php`, `cook/orders.php`.
* **Backend Functionality:** Creates authentic rural recipes and receives food orders from Tourists.
* **Database Connection:** `recipes.php` inserts into the `RECIPE` table. `orders.php` queries `FOOD_ORDER` where `cook_id` matches their ID, allowing them to UPDATE the order status to `'delivered'`.

### 6. 🔬 Expert
* **Frontend:** `expert/dashboard.php`, `expert/sessions.php`.
* **Backend Functionality:** Receives paid consultation requests regarding crop health, soil, and diseases. Can chat with clients.
* **Database Connection:** `sessions.php` queries `CONSULTATION` where `provider_id` equals the expert's user ID. Interacts with `CONSULTATION_MESSAGE` in the chat module.

### 7. 🗺️ Guide
* **Frontend:** `guide/dashboard.php`, `guide/bookings.php`, `guide/sessions.php`.
* **Backend Functionality:** Receives requests to guide Tourists on farm tours. Also receives generic consultation requests for tourism advice.
* **Database Connection:** Queries `TOUR_BOOKING` where `guide_id` matches their ID. Queries `CONSULTATION` for advisory sessions.

### 8. 🏬 Dealer
* **Frontend:** `dealer/dashboard.php`, `dealer/inventory.php`, `dealer/sales.php`.
* **Backend Functionality:** Represents Bulk B2B buyers. (Originally the middleman for consumers, but the marketplace is now Direct-to-Consumer). Dealers manage their own secondary inventory logic.
* **Database Connection:** Interacts with the `DEALER_INVENTORY` table.
